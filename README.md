# Dunner

**The only failed-payment recovery tool that sounds like you.**

When a SaaS subscription fails, Dunner calls the customer in the merchant's own
cloned voice, negotiates a recovery in real-time using live Stripe Subscriptions
API access, and earns a percentage only when the recovery succeeds.

iOS-first (Expo), production-grade infrastructure from day one.

> 🚧 Hackathon build — ElevenLabs Hack #9, Stripe sponsor track.

## Architecture

```
┌──────────────────────────────────────────────────────────────────────┐
│                          MERCHANT (Mobile)                           │
│  Expo SDK 55 · NativeWind · Reanimated 4 · Skia · Clerk · WS         │
└──────────────────────────────────────────────────────────────────────┘
                              │ HTTPS / WSS
                              ▼
┌──────────────────────────────────────────────────────────────────────┐
│                       BACKEND (Hono on Bun · Railway)                │
│  REST API ──────── Webhooks ─────── WS hub ─────── Agent tools       │
│  /me /recoveries  /webhooks/stripe  /ws/call/:id  /stripe-actions/*  │
│  /agent/config    /webhooks/eleven  /ws/merchant  (bearer-auth)      │
│  /admin/* …       …                 …             …                  │
└──────────────────────────────────────────────────────────────────────┘
        │              │              │              │
        ▼              ▼              ▼              ▼
   Postgres        Stripe         Upstash       ElevenLabs
   (Drizzle)      Connect +       Redis         ConvAI + Telnyx
   on Railway     Subscriptions   (ratelimit)   (SIP trunk)
```

### Recovery flow

1. Merchant's Stripe account fires `invoice.payment_failed`
2. Backend webhook verifies signature, dedups, persists a `recoveries` row
   in `QUEUED` state, sets `application_fee_amount` on the PaymentIntent
3. Scheduler tick respects merchant's working hours + max retries, calls
   ElevenLabs `/v1/convai/twilio/outbound-call` with the merchant's cloned
   voice and dynamic variables
4. ElevenLabs places the call via Telnyx SIP trunk in the merchant's voice
5. During the call, the agent may invoke tools — `pause_subscription`,
   `apply_coupon`, `swap_payment_method`, `downgrade_plan`,
   `send_recovery_link`, `log_callback`, `log_churn`, `end_call`
6. Each tool fires a webhook to `/stripe-actions/*` which uses Stripe's
   Connect API to act on the merchant's account, and broadcasts a WS event
   to the mobile app
7. Call ends → `post_call_transcription` webhook fires → backend reads
   `tool_calls_fired`, transitions recovery state, broadcasts call.ended
8. When the customer eventually pays via fresh link / new PM, Stripe fires
   `invoice.paid` → backend marks recovery `RECOVERED`, fee already captured

See [CLAUDE.md](./CLAUDE.md) for the full state machine, schema, and
gotchas.

## Local development

### Prerequisites

- **Bun** `≥ 1.3.14` ([bun.sh](https://bun.sh))
- **Node.js** `≥ 20.9.0` (for landing's Next.js)
- **pnpm** `≥ 10` (`npm i -g pnpm`)
- **Xcode 16+** with iOS 18 simulator (for mobile)
- **Stripe CLI** (`stripe listen` for webhook forwarding)
- **Railway CLI** (`railway` — optional, for deploys)
- **Postgres** — either Railway, Supabase, or local

### First run

```bash
# Install deps for every workspace
cd backend && pnpm install
cd ../landing && npm install
cd .. && pnpm install   # root = mobile

# Copy env templates
cp backend/.env.example  backend/.env.local
cp landing/.env.example  landing/.env.local
cp .env.example          .env.local           # mobile

# Fill in keys (see "Environment variables" below)

# Run database migrations
cd backend && pnpm drizzle-kit migrate

# Start everything (in separate terminals)
bun run --watch src/index.ts            # backend  → :3000
cd ../landing && npm run dev            # landing  → :3001
cd .. && pnpm expo run:ios              # mobile   → iOS sim

# Forward Stripe webhooks
stripe listen --forward-to http://localhost:3000/webhooks/stripe
```

## Deploying

### Backend → Railway

```bash
cd backend
railway up
```

Bun is autodetected by Railpack. Start command: `bun run src/db/migrate.ts && bun --smol run src/index.ts`. Set every env var from the backend table above in the Railway service.

### Landing → Railway

```bash
cd landing
railway up
```

Set root directory to `landing/` in the Railway service settings.
Add the env vars from the landing table. Link `dunner.xyz` in
Railway → Settings → Domains. Wait for cert provisioning (~2 min).

### Mobile → EAS

```bash
# Development build (for your phone)
pnpm dlx eas-cli build --profile development --platform ios
pnpm dlx eas-cli build --profile development --platform android

# Production
pnpm dlx eas-cli build --profile production --platform android
# (iOS requires Apple Developer account for TestFlight)
```

## Admin dashboard

Visit `/admin` on the landing site. Auth flow:

1. Clerk-authenticate with any email in `ADMIN_EMAILS`
2. The dashboard fetches from `BACKEND_URL/admin/*` with your Clerk JWT
3. Both the landing edge and the backend re-verify the allowlist
   (defense in depth — the backend won't trust the landing's check alone)

Routes:

| Path                  | What it shows                                                  |
| --------------------- | -------------------------------------------------------------- |
| `/admin`              | KPIs: merchants, waitlist, recoveries, total $ recovered       |
| `/admin/waitlist`     | All signups · invite (send code) · remove                      |
| `/admin/merchants`    | Onboarding progress per merchant                               |
| `/admin/recoveries`   | Recent recoveries with customer + amount + state               |

## Security model

- **Clerk JWT** verified networkless via `verifyToken` (JWKS cached at startup)
- **Stripe + ElevenLabs webhooks** signature-verified BEFORE any DB writes
- **Webhook idempotency**: Postgres UNIQUE constraint on `event_id` +
  optional Redis SETNX layer for hot dedup
- **Per-merchant scoping**: every authed query filters by the resolved
  `merchant_id`; one merchant cannot see another's recoveries
- **Agent callbacks** (`/stripe-actions/*`): bearer token bound to a single
  merchant, NOT Clerk JWT. Stored hashed in `agent_api_tokens`
- **Application fee** set on the PaymentIntent BEFORE the recovery payment,
  passed with `{ stripeAccount: connectedAccountId }` — fees flow to
  platform balance automatically
- **Public endpoints** (`/waitlist`, `/auth/redeem-code`) rate-limited by IP
  via Upstash sliding window
- **Email enumeration suppressed** — `POST /waitlist` returns identical
  shape regardless of whether the row existed
- **Admin allowlist** enforced at both landing edge and backend
